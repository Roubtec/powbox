# Entrypoint and runtime internals

Agent-facing reference for container startup: the entrypoint chain, the per-agent seeding hooks, the ordering constraints between them, host-workspace permission healing, the mid-session pnpm/shadow wrapper, and the deliberate bash/zsh split.
Routed from [AGENTS.md](../AGENTS.md); see [architecture.md](architecture.md) for the image/identity/volume picture this startup sequence sits inside.

## Entrypoint and Runtime

- `entrypoint-agent.sh` is the image ENTRYPOINT. It reads `PRIMARY_AGENT` (`claude` | `codex`, defaulting to `claude` for an unknown value), holds the agent registry (`agent_env` maps each agent to its `AGENT_CONFIG_DIR`, `AGENT_SETUP_HOOK`, `AGENT_SEED_DIR`, name, binary, autonomy flag, instruction file, and label), then seeds every agent and execs `entrypoint-core.sh` for the primary. Adding a harness = extend `agent_env` and `ALL_AGENTS`.
- It seeds in two passes: each non-primary agent is seeded directly (export its `AGENT_*` vars, run its hook); the primary agent's env is exported and handed to `entrypoint-core.sh`, which runs the primary's hook (so it is not run twice) alongside firewall/git/shadow setup before execing the CMD.
- Each per-agent hook (`entrypoint-claude-hook.sh`, `entrypoint-codex-hook.sh`) is run in full for every agent. Each writes only into its own config dir (`~/.claude` vs `~/.codex`/`~/.agents`), so there is no conflict; hooks are idempotent, no-clobber, and build-epoch-gated, and read their baked assets from `AGENT_SEED_DIR`.
- `entrypoint-core.sh` is a wrapper-style entrypoint that must end with `exec "$@"` and is unchanged by the unification (still runs the single `AGENT_SETUP_HOOK`).
- The shared instruction template is rendered via `envsubst` with agent-specific variables (including `${AGENT_PEERS}`, the registry-derived peer list for the "Delegating to another agent" section).
- `gh auth setup-git` runs from `$HOME` (not the workspace) and failure is non-fatal. On success it also adds a container-global `url."https://github.com/".insteadOf "git@github.com:"` rewrite (written to the ephemeral `GIT_CONFIG_GLOBAL`, never the host) so SSH-form `origin` remotes push/fetch over HTTPS+gh without rewriting the host repo.
- Before the git/`safe.directory` steps, `entrypoint-core.sh` write-probes each dir-mounted `/workspace/<slug>` as `node` and, for any it cannot write, `sudo`s `fix-workspace-perms.sh`, which `chown`s it to `node` when the workspace root is owned by `root` (uid 0) — the common native-Linux case (a repo under `/root`, or a host running powbox as root), where the agent otherwise gets EACCES on every `git`/edit. The write probe alone misses a **mixed-ownership** workspace — a *node-owned* root that nonetheless hides nested `root`-owned files, which arises when a host operation runs as root against a live bind mount (most often `sudo git pull`, which re-owns to uid 0 only the paths it writes — new `.git/objects/*`, refs, changed working-tree files — leaving the top dir node-owned), so the agent can write the root yet `git commit`/`add` still fail with `insufficient permission for adding an object to repository database`. So besides the probe, the entrypoint runs a short-circuiting `find … -uid 0 -print -quit` (with the same mountpoint prune as below) over each node-writable workspace and hands any that contains a uid-0 entry to the same helper; the helper now acts on a root owned by **root or node** and re-owns only the `find -uid 0` entries (for a root-owned root that is the whole tree; for a node-owned root just the nested uid-0 entries — the root, already node, is untouched). A workspace whose root is owned by some *other* non-root, non-node host uid is refused with remedies (chown the repo to uid 1000 on the host, or relaunch in `--isolated` mode), never silently chowned, so a real host user is never locked out of their own repo. The probe creates its temp file with `mktemp` (O_EXCL, random name) rather than a predictable path, so a planted symlink in an untrusted workspace can't make it truncate an outside target as node. It is probe-/scan-gated, so it no-ops on Windows/WSL (FUSE bind mounts already honour node's writes) and on uid-matched hosts with no nested uid-0 entries; self-hosted mode and the detached image-store writer are exempt. The helper canonicalizes each argument and only acts on a direct `/workspace/<slug>` child; the chown (and the entrypoint's detection scan) prunes the real mountpoints nested under the workspace (from `/proc/self/mountinfo`), with `find -xdev` only as a backstop, so it never descends into the nested `node_modules`/`.worktrees` volume mounts — `-xdev` alone is insufficient when a volume shares the bind-mount source's filesystem (same st_dev). The recursive chown matches only root-owned entries (`find -uid 0`), so a file or directory nested inside a claimed tree but owned by another host uid keeps its owner rather than being re-owned to node — the same protection the root-level refusal gives, applied per-entry.
- **Sensitive-source refusal.** Before any of the above, both the heal decision (`heal-workspace-perms.sh`) and the privileged helper (`fix-workspace-perms.sh`) refuse outright to chown a workspace whose **host bind-mount source** is a system or home directory — `/`, `/root`, `/home/<user>`, `/Users/<user>`, the launching user's `$HOME`, or a bare FHS root like `/etc`/`/usr`/`/var`/`/opt`, plus any SSH config directory (a path ending in `/.ssh`, e.g. `/home/alice/.ssh` or `/root/.ssh`) — chowning just `~/.ssh` and its `authorized_keys` to node is itself enough to trip StrictModes even when the home dir stays root-owned (a project *nested* under a system/home root — `/opt/app`, `/var/www/html`, `/home/alice/code/app` — still heals normally). This guards a real incident: a `cc`/`cx` accidentally launched from `~` (e.g. `/root` on a VPS) bind-mounts the whole home tree as the "project", and the heal would then recursively chown it to node — re-owning `~/.ssh` **and its parent**, which breaks OpenSSH's StrictModes ownership chain so `sshd` refuses `~/.ssh/authorized_keys` and locks you out of the host (recoverable only via an out-of-band console). When the source is sensitive the workspace is **mounted but not chowned**, with a loud warning, so an accidental launch is a recoverable inconvenience (node may be unable to write that tree) instead of a host lockout. The shared predicate lives in `sensitive-host-path.sh` (sourced by both). `heal` consults the launcher-provided `POWBOX_WORKSPACE_HOST_PATH`/`POWBOX_WORKSPACE_HOST_HOME` env (authoritative, and the only reliable signal on Docker Desktop/WSL) **and** `/proc/self/mountinfo`; `fix`, running under sudo (`env_reset` strips the env), trusts only `/proc/self/mountinfo` (field 4 — the bind source, which an unprivileged `node` cannot forge) as a **best-effort** independent backstop when invoked directly — best-effort because mountinfo field 4 is the mount root *within its source filesystem*, not the absolute host path, so on a separate-mount layout (e.g. a dedicated `/home`) a `/home/alice` bind reads back as `/alice` and can slip past it (feeding both checks the launcher's true source, with mountinfo as fallback, is tracked in `tasks/009`). The predicate and the mountinfo lookup are unit-tested by `scripts/test-sensitive-host-path.sh`; the live heal-skip and fix-refusal are smoke-tested by the dir-mount stage.
- Self-hosted (`--isolated`) clone seeding runs **after** the `gh auth` step (so a private clone has credentials) and **before** the shadow/pnpm steps, via the base-baked `seed-workspace.sh` (gated on `POWBOX_SELF_HOSTED=1`; a no-op otherwise, including the image-store writer role). It clones `POWBOX_CLONE_REPO`'s default branch into `POWBOX_WORKSPACE_DIR`, then applies the optional `POWBOX_CLONE_REF` as a **post-clone `git checkout`** (so it accepts a branch, tag, or raw commit SHA, and an unresolvable ref degrades to a warning + stay-on-default rather than aborting the clone), and skips the whole thing when a `.git` already exists (reuse — the agent owns its branches/worktrees; the ref is not re-applied on reuse). `--reclone` is **not** a frozen env flag (that would re-wipe the agent's work on every restart): it is a one-shot launcher action that empties the kept `agent-ws-*` volume and recreates the container, so the entrypoint then sees an empty dir and clones fresh. There is deliberately **no clone/auth failsafe and no retry**: on any failure it prints a loud announcement of the three remedies (use dir-mounted mode / `gh auth login` here / seed the shared gh volume) and `entrypoint-core.sh` `exec`s a plain `zsh` rather than execing the agent into an empty workspace.
- `safe.directory`/`core.filemode` handling is left as-is (harmless on a node-owned clone).
- Workspace shadow mounts run after git setup, so any shadow logic must not assume an earlier ordering. `shadow-mounts.sh` is **skipped** in self-hosted mode (no host FS to shadow, and a tmpfs would break the single-mount hardlinking); `wt-bootstrap` likewise branches its root-safety check on `POWBOX_SELF_HOSTED` (the worktree roots are plain subdirs of the one workspace volume, not per-dir mounts).
- Startup shadow detection only sees the subpackages that exist at container start, so a package scaffolded mid-session would have its `node_modules` populated straight onto the host bind mount before any re-shadow. To close that race, `pnpm`/`pn` are replaced (base image, after `npm install -g pnpm`) with `docker/shared/pnpm-shadow-wrapper.sh`: for node_modules-writing subcommands (`install`/`add`/`update`/… — detected by scanning all args, since global flags can precede the subcommand) it runs `shadow-refresh.sh` scoped to the `/workspace/<slug>` enclosing pnpm's *effective* directory — honoring `-C, --dir <dir>`, so a `pnpm -C /workspace/<repo> install` invoked from outside `/workspace` still re-shadows the right project — *before* exec'ing the real `/usr/local/lib/node_modules/pnpm/bin/pnpm.mjs`. It is a no-op when that effective directory is outside `/workspace` and in self-hosted mode (`POWBOX_SELF_HOSTED`), mirroring `shadow-mounts.sh`, and always exec's pnpm so a shadow failure never blocks the command. `pnpx`/`pnx` (dlx) are left unwrapped — they don't write workspace `node_modules`. The same wrapper also closes the launch-time gate's residual hole (task 002b): a folder launched as **non-dev** (no `package.json`/`pnpm-workspace.yaml`/`.powbox.yml`, so the launcher mounts no `agent-nm-*` root volume and sets no `PNPM_STORE_DIR`) that gains a `package.json` mid-session would, on a **root** install, write `node_modules` straight onto the host bind mount — the very host litter the gate exists to prevent, and which the wrapper cannot retrofit a mount for. So before such an install it prints exactly one loud stderr warning (then still proceeds) telling the user to relaunch the agent (the folder now has a `package.json`, so the next launch mounts an isolated volume). The regression is detected via `<workspace>/node_modules` not being a mountpoint **plus** an empty `PNPM_STORE_DIR` (the launcher sets it only when it mounted the volume) and the effective dir being the workspace root, so a self-hosted run, an already-mounted dir-mounted dev project, and a subpackage install never trip it; it is unit-tested by `scripts/test-pnpm-shadow-wrapper.sh`.
- The container's two shells are split on purpose, so the `usermod --shell /bin/zsh` and `ENV SHELL=/bin/bash` in `docker/base/Dockerfile` are not a contradiction: zsh is the human login shell (and `launch-agent --shell` execs `zsh` directly), while bash is the `$SHELL` the agent harnesses' Bash tools spawn — the models write bash/POSIX (word-splitting, 0-indexed arrays) and a non-interactive one-shot call gains nothing from zsh, so defaulting agents to zsh only wasted turns. Both rc files (`docker/shared/.bashrc`, `.zshrc`) are baked but only shape interactive sessions; `EDITOR` and `PATH` live in image `ENV` so neither shell owns them.
